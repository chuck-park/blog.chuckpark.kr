# [AI로 서비스 만들기] PG 연동 전 꼭 닫아야 할 구멍: Email 가입/로그인 + 비밀번호 재설정

## 한 줄 요약
소셜 로그인만으로는 결제 전환 퍼널이 닫히지 않는다. 카카오·구글 계정이 없는 사용자도 들어올 수 있도록 Email 가입/로그인과 비밀번호 재설정 흐름을 먼저 완성했다.

## 왜 이 작업이 PG 연동과 연결되는가
이번 시리즈의 핵심은 “AI 서비스가 실제 결제 가능한 제품이 되도록 만드는 과정”이다.  
PG 연동만 붙이면 결제가 완성되는 것 같지만, 로그인 진입점이 좁으면 결제 단계까지 도달하지 못하는 사용자가 생긴다.

기존 호랑사주는 카카오/Google 소셜 로그인만 지원했다.  
문제는 명확했다.

- 소셜 계정 사용이 어려운 사용자 이탈
- 계정 복구 수단 부재(비밀번호 찾기 없음)
- 결제 전에 인증 단계에서 전환 손실

그래서 이번 챕터에서는 인증 퍼널을 먼저 넓혔다.

---

## 1) 로그인 페이지 UI 변경

### 변경 전
- 카카오로 로그인
- Google로 로그인

### 변경 후
- 카카오로 시작하기
- Google로 시작하기
- Email로 시작하기

핵심은 문구였다.  
`로그인`은 이미 계정이 있는 사람의 행동처럼 느껴진다. 신규 유저에게는 진입 장벽이 된다.  
`시작하기`는 가입과 로그인을 모두 포함하는 표현이라 첫 클릭 저항이 낮다.

또한 Email 인증은 별도 페이지로 빼지 않고 같은 카드 안에서 로그인/회원가입 토글로 처리했다.  
페이지 이동은 생각보다 큰 이탈 포인트다.

---

## 2) Email 회원가입 API 설계

`POST /api/auth/register`

구현 원칙은 단순했다. “가입 성공까지의 단계는 최소화한다.”

- 이메일 형식 검증
- 비밀번호 6자 이상 검증
- 중복 이메일이면 `409` 반환
- `bcrypt`로 해시 후 User 생성
- 가입 즉시 자동 로그인

초기 단계에서는 이메일 인증 단계를 일부러 넣지 않았다.  
퍼널 초반에 검증 단계를 추가하면 이탈이 늘어난다. 필요하면 사용자 규모가 커질 때 도입하면 된다.

기술적으로는 기존 스택을 재사용했다.

- `bcryptjs` 이미 사용 중
- Auth.js Credentials Provider 이미 구성됨

새로운 인증 인프라를 도입하지 않고 기능을 확장할 수 있었다.

---

## 3) 비밀번호 찾기 구현

### 3-1. 이메일 발송 서비스: Resend 선택

소규모 서비스 기준에서 Resend가 가장 현실적이었다.

- 무료 한도: 월 3,000건
- API 키 한 개로 시작 가능
- SDK 단순 (`resend.emails.send()`)

비밀번호 재설정 메일은 월 수십 건 수준이라 무료 플랜으로 충분하다.  
초기에는 “유지 가능한 단순성”이 성능 못지않게 중요했다.

### 3-2. 토큰 전략: JWT 대신 HMAC 서명 토큰

비밀번호 재설정 토큰을 위해 별도 DB 테이블을 만들지 않았다.  
대신 `AUTH_SECRET` 기반 HMAC-SHA256 서명 토큰을 사용했다.

구조:

`{ email, exp } -> base64url -> HMAC 서명 -> data.sig`

장점:
- DB 마이그레이션 불필요
- 토큰 저장/정리 배치 불필요
- 검증 로직 단순

트레이드오프:
- 개별 토큰 강제 폐기 어려움

하지만 재설정 토큰은 1시간 만료, 단기/단발성 용도라 초기 서비스에서는 실용적인 선택이었다.

### 3-3. 보안 포인트: 이메일 존재 여부 비노출

`forgot-password` API는 가입되지 않은 이메일이어도 동일 응답을 반환한다.

- “이메일을 확인해주세요” 고정 응답

이메일 존재 여부를 에러 메시지로 드러내면 계정 열거 공격에 악용될 수 있다.

---

## 4) 사용자 플로우 정리

### 회원가입
Email 버튼 → 회원가입 탭 → 이메일/비밀번호 입력 → 가입 → 자동 로그인 → 프로필 설정

### 비밀번호 찾기
Email 버튼 → “비밀번호를 잊으셨나요?” → 이메일 입력 → 재설정 링크 수신 → 새 비밀번호 입력 → 로그인 페이지 이동

---

## 5) 추가된 파일

- `src/app/api/auth/register/route.ts` — 회원가입 API
- `src/app/api/auth/forgot-password/route.ts` — 재설정 링크 발송 API
- `src/app/api/auth/reset-password/route.ts` — 비밀번호 변경 API
- `src/app/reset-password/page.tsx` — 새 비밀번호 입력 페이지
- `src/lib/email.ts` — Resend 연동
- `src/lib/reset-token.ts` — HMAC 토큰 생성/검증

---

## 마무리: 이번 단계의 원칙
이번 작업의 기준은 하나였다.

> “최소한의 인프라로, 실제로 동작하는 인증 퍼널을 먼저 완성한다.”

DB 구조를 크게 바꾸지 않고, 환경변수(`RESEND_API_KEY`) 추가 수준으로 기능을 닫았다.  
이메일 인증, 비밀번호 강도 UI, 로그인 시도 제한 같은 고급 기능은 사용자 증가 시점에 붙여도 늦지 않다.

PG 연동은 결제의 시작점이지만, 그 전에 인증 퍼널이 막히면 결제까지 도달하지 못한다.  
그래서 이번 챕터에서는 결제 이전의 병목부터 먼저 정리했다.
