# [AI로 서비스 만들기] 결제 붙이기의 진짜 시작, 카드사 심사 단계 기록

## 한 줄 요약
PG사 연동이 끝나도 결제 오픈은 아직 아니다. 실제 오픈 시점을 결정하는 건 카드사 심사 진행 상태를 얼마나 정확하게 추적하느냐다.

## 카드사 심사 전에 먼저 여는 단계: KCP PG-API 발급
실제 결제 시스템을 붙이려면 카드사 심사 이전에 먼저 통과해야 하는 관문이 있다. 바로 KCP PG-API 발급이다.

이 단계는 버튼 한 번으로 끝나지 않는다. **`미리 문의하기`로 사전 신청하고, 확인이 완료되어야** API 발급/재발급 절차로 넘어갈 수 있다.

즉, 개발 순서로 보면 결제 코드 작성보다 앞단에서 운영 승인 흐름을 먼저 통과해야 한다.

## PG 승인 후에 진짜로 시작되는 일
결제 연동 작업을 처음 할 때 가장 많이 착각하는 지점이 있다.

> “PG 연결 완료 = 결제 오픈 준비 완료”

실무에서는 다르다. PG사 내부 검토가 끝난 뒤에도 카드사 심사가 남아 있고, 이 단계에서 오픈 일정이 밀리거나 보완 요청이 들어올 수 있다. 그래서 이 구간부터는 개발보다 운영 대응이 중요해진다.

## 카드사 심사 화면에서 먼저 보는 것
카드사 심사 등록현황 화면에서 내가 가장 먼저 보는 건 화려한 지표가 아니다. 아래 세 가지다.

1. 카드사별 상태값
2. 최근 업데이트 시간
3. 비고/요청 사항 유무

이 세 가지가 명확하지 않으면 “심사가 어디서 멈췄는지”를 판단하기 어렵다.

## 상태값 해석을 운영 언어로 바꾸기
상태 텍스트를 그대로 읽기만 하면 대응이 늦어진다. 운영 관점으로 해석해야 한다.

- **심사중**: 대기 구간. 일정 버퍼를 확보하고, 추가 요청 대비 자료를 정리해 둔다.
- **완료**: 해당 카드사는 통과. 전체 오픈 기준(주요 카드사 포함 여부)을 체크한다.
- **보류/반려/추가요청**: 즉시 대응 구간. 미루면 전체 오픈 일정이 바로 밀린다.

핵심은 “전체 완료”만 보는 게 아니라, 실제 고객 결제 비중이 높은 카드사 상태를 우선순위로 추적하는 것이다.

## 심사 지연 시 대응 순서
내가 쓰는 기본 대응 순서는 단순하다.

1. **현재 상태 확정**: 어떤 카드사에서 멈췄는지 먼저 명확히 한다.
2. **요청사항 확인**: 보완 요청이 있으면 텍스트를 운영 태스크로 분해한다.
3. **증빙/화면 즉시 보완**: 사업자 정보, 이용약관, 환불/고객센터 정보 등 누락 여부 재점검.
4. **재전달 + 확인 시점 고정**: “언제 다시 확인할지”를 명시해 공회전을 막는다.

여기서 중요한 건 감으로 기다리지 않는 것이다. 확인 시점을 못 박아야 일정이 관리된다.

## 오픈 전 최종 체크리스트
카드사 심사가 거의 끝났을 때, 아래를 마지막으로 점검한다.

- 주요 카드사 결제 가능 상태 확인
- 테스트 결제/취소 플로우 재검증
- 실패 응답 처리(에러 메시지, 재시도 UX) 점검
- 정산/취소/환불 운영 담당자 동선 확인

개발은 끝났는데 운영이 준비 안 된 상태로 오픈하면, 실제 이슈는 오픈 이후에 터진다.

## 마무리
결제 붙이기의 진짜 난이도는 API 연결보다 “심사 상태를 운영 가능한 정보로 바꾸는 과정”에 있다.

이번 단계에서 배운 건 하나다.

**심사 상태를 읽는 속도가 곧 오픈 속도다.**
