# Inkframe Theme Productization Plan

목표는 현재 블로그에 적용한 `Inkframe` 커스텀 레이어를 재사용 가능한 Jekyll 테마 상품으로 분리하고, 문서와 데모를 갖춘 판매 가능한 패키지로 만드는 것이다.

## 목표

- 현재 블로그 커스텀 UI를 `Inkframe`라는 이름의 독립 테마 자산으로 정리한다.
- 설치 문서와 데모 사이트를 포함한 첫 판매 버전을 만든다.
- 초기 판매 채널은 Gumroad를 기준으로 준비한다.

## 제품 정의

`Inkframe`는 `minimal-mistakes` 위에 올린 커스텀 프론트 레이어가 아니라, 설치 가능한 별도 Jekyll theme product로 다시 패키징한다.

초기 버전에 포함할 범위:

- 홈/아카이브 리스트 레이아웃
- 포스트 상세 레이아웃
- 다크/라이트 테마 토글
- 모바일 햄버거 메뉴
- 카테고리 상세 페이지 구조
- Cusdis, Buttondown 연동 가이드

초기 버전에서 제외할 것:

- 사용자별 맞춤 커스터마이징 서비스
- CMS 연동
- 다중 스킨 번들
- 유료 플러그인 번들

## 판매 전략

초기 판매 채널은 Gumroad를 우선한다.

이유:

- 심사 부담이 낮다.
- ZIP, gem, 문서, 예제 파일을 함께 배포하기 쉽다.
- 업데이트 파일 교체와 구매자 관리가 간단하다.

판매 포지셔닝:

- 대상: Jekyll 또는 GitHub Pages 기반 개인 블로그 운영자
- 가치: 기본 Minimal Mistakes보다 더 현대적인 블로그 경험을 빠르게 적용
- 강점: 다크/라이트 전환, 카드형 리스트, 콘텐츠 중심 상세 페이지, 구독/댓글 연동 경험

## 배포 형태

권장 배포 형태는 두 단계다.

### 1단계: ZIP 기반 판매 패키지

포함물:

- `_layouts`, `_includes`, `_sass`, `_data` 등 테마 핵심 파일
- 설치 가이드
- 샘플 `_config.yml`
- 샘플 포스트
- 변경 포인트 문서

장점:

- 가장 빨리 판매 시작 가능
- GitHub Pages 사용자가 이해하기 쉽다

단점:

- 업그레이드 경험이 거칠다
- 코드 재사용성이 낮다

### 2단계: gem-based theme

Jekyll 공식 테마 구조로 옮겨 RubyGems 배포 또는 private repo 배포를 준비한다.

장점:

- 설치와 업데이트 경험이 좋아진다
- 문서 구조가 표준화된다

단점:

- 패키징과 테스트 비용이 늘어난다
- GitHub Pages와의 호환 안내가 더 중요해진다

권장 순서:

1. ZIP 판매 버전으로 시장 반응 확인
2. 반응이 좋으면 gem 기반으로 재구성

## 작업 구조

### 트랙 1: 코드 분리

- 현재 블로그 repo에서 `Inkframe` 레이어 파일 목록을 확정한다.
- `minimal-mistakes` 의존과 직접 구현 부분을 나눈다.
- 테마 전용 repo 또는 패키지 디렉터리로 분리한다.

핵심 후보 파일:

- `_layouts/archive.html`
- `_layouts/home.html`
- `_layouts/single.html`
- `_includes/masthead.html`
- `_includes/footer.html`
- `_includes/head/custom.html`
- `_includes/footer/custom.html`
- `_includes/archive-single.html`
- `_includes/post-subscribe.html`
- `_includes/comments-providers/custom_scripts.html`
- `_sass/custom.scss`
- `_data/navigation.yml`
- `_data/ui-text.yml`

### 트랙 2: 설정 표준화

- `_config.yml`에서 사용자마다 바뀌는 값을 분리한다.
- 사이트명, 설명, 네비게이션, Cusdis, Buttondown, 소셜 링크를 문서화한다.
- 필수 설정과 선택 설정을 나눈다.

예상 설정 그룹:

- 브랜딩
- 메뉴
- 포스트 리스트 옵션
- 댓글 시스템
- 구독 시스템
- 다크/라이트 토글

### 트랙 3: 문서화

최소 문서 세트:

- 설치 가이드
- 설정 가이드
- 커스터마이징 가이드
- 배포 가이드
- FAQ

문서에서 반드시 다룰 것:

- GitHub Pages 사용 시 제약
- Cusdis/Buttondown 연결법
- 라이트/다크 토글 동작 방식
- 썸네일, 카테고리, 관련 글 구조

### 트랙 4: 데모 사이트

데모 사이트 목표:

- 실제 리스트 화면
- 실제 포스트 상세 화면
- 카테고리 화면
- 다크/라이트 전환
- 모바일 메뉴

데모는 상품 페이지보다 먼저 준비한다. 데모가 있어야 구매 설득이 된다.

## 가격 전략

초기 가격 가설:

- 개인용 단일 라이선스: 49~79달러

후속 옵션:

- 멀티사이트 라이선스
- 업데이트 포함 기간 명시
- 설치 지원 포함 버전

초기에는 복잡한 가격표보다 단일 가격이 낫다. 가장 먼저 확인할 것은 “설치 가능한 Jekyll 블로그 테마에 돈을 내는 수요가 있는가”다.

## 판매 페이지 구성

필수 섹션:

- 무엇을 해결하는가
- 어떤 화면이 제공되는가
- 누구에게 맞는가
- 포함 파일
- 설치 난이도
- 지원 범위
- 데모 링크
- 스크린샷

피해야 할 것:

- 지나치게 많은 옵션 나열
- “모든 블로그에 맞는다” 같은 과장
- GitHub Pages 호환성에 대한 모호한 표현

## 출시 순서

### Phase 1: 상품화 준비

- `Inkframe` 레이어 파일 고정
- 설정 포인트 분리
- 설치 문서 작성
- 데모 사이트 작성

### Phase 2: 첫 판매 버전

- Gumroad 상품 페이지 오픈
- 데모 링크 공개
- 예제 ZIP 배포
- 구매 후 설치 흐름 점검

### Phase 3: 개선

- 자주 묻는 질문 정리
- 설치 이슈 수집
- gem 기반 전환 여부 판단
- 추가 스킨 또는 스타터 팩 검토

## 리스크

### 기술 리스크

- `minimal-mistakes` 업데이트에 따라 override 구조가 깨질 수 있다.
- GitHub Pages 환경에서 gem theme 배포 경험이 제한될 수 있다.

### 상품 리스크

- 설치 난이도가 높으면 환불과 지원 부담이 커진다.
- 문서 품질이 낮으면 코드 품질과 무관하게 상품 가치가 떨어진다.

### 운영 리스크

- 댓글/구독 같은 외부 서비스가 바뀌면 문서가 빨리 낡을 수 있다.
- 커스터마이징 문의가 많아지면 사실상 서비스 사업이 될 수 있다.

## 추천 다음 단계

1. 현재 블로그에서 `Inkframe` 관련 파일만 추려 별도 체크리스트를 만든다.
2. ZIP 기반 첫 판매 버전을 목표로 설치 문서와 샘플 설정을 만든다.
3. 데모 사이트를 먼저 공개하고, 이후 Gumroad 상품 페이지를 연다.
4. 초기 구매자 3~5명의 설치 피드백을 받은 뒤 gem 기반 전환 여부를 결정한다.

## 성공 기준

- 데모만 보고 테마의 가치가 이해된다.
- 설치 문서만으로 새 블로그에 적용할 수 있다.
- 외부 서비스 연결이 문서만으로 가능하다.
- 첫 판매 후 반복 문의 없이 설치 완료 사례가 나온다.
